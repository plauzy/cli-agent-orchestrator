# Antigravity CLI Provider

CAO provider for the Antigravity IDE's `agy` headless CLI (Google's coding-agent CLI, built on Gemini 3.x, shipped alongside the Antigravity IDE).

Status: **chat-only v0**. The provider can launch `agy`, drive an interactive session, and extract responses. MCP server registration is **deferred to v0.1** — see [MCP support](#mcp-support-deferred-to-v01) below for the rationale.

## Installation

Antigravity CLI is distributed as a single static Go binary, typically installed at `~/.local/bin/agy`. Verify with:

```bash
agy --version   # → 1.0.6 (or later)
agy --help
```

If `agy` is not on `PATH`, run `agy install` once after placing the binary to wire up shell paths.

## Configuration

Provider configuration lives in the CAO agent profile (markdown with YAML frontmatter under `agent_store/`). The provider recognises the standard `name`, `role`, `description`, `model`, and `system_prompt` fields.

- `model` — passed to `agy --model "<name>"`. Names are human-readable (e.g. `"Gemini 3.1 Pro (High)"`); `agy models` lists what the installed version accepts.
- `system_prompt` — passed via `agy -i "<text>"` (`--prompt-interactive`). Sent as the first user message; the model adopts it as a role brief.
- `allowedTools` — currently a no-op for `agy` (see [Tool restrictions](#tool-restrictions)).

`agy`'s own state directory is `~/.gemini/antigravity-cli/` (shared parent with `gemini` CLI). The provider does **not** write into this directory in v0.

## MCP support (deferred to v0.1)

CAO v2.5 does **not** register MCP servers when launching `agy`. The provider intentionally runs `agy` in chat-only v0 mode. If an agent profile declares `mcpServers`, the provider logs a warning and proceeds without registration.

### Why deferred

Investigation (`agy 1.0.6`, macOS arm64, 2026-06-08) found that `agy` does **not** expose a stable, documented surface for registering MCP servers programmatically the way `gemini` does:

| Surface | Status |
|---|---|
| `agy mcp add` / `mcp` subcommand | **Does not exist.** `agy`'s top-level subcommands are `changelog`, `help`, `install`, `models`, `plugin`, `plugins`, `update` — none of them target MCP servers directly. |
| Direct `mcpServers` key in `~/.gemini/antigravity-cli/settings.json` | **Not honored.** Empirical test: writing a synthetic `mcpServers` entry into agy's settings.json and asking the running model to enumerate its tools did not surface the test server. |
| Direct `mcpServers` key in shared `~/.gemini/settings.json` | Honored by `gemini` CLI but not by `agy`; the two CLIs share the parent directory but read distinct configs. |
| Plugin-based registration via `~/.gemini/antigravity-cli/plugins/<name>/{plugin.json,mcp_config.json}` + `import_manifest.json` entry | **Plumbing exists** (existing `google-workspace`, `gemini-cli-security` plugins use exactly this shape), **but** the `import_manifest.json` is regenerated by `agy plugin import`, which only accepts canonical sources (`gemini`, `claude`) — hand-written entries are silently dropped by `agy plugin list`, indicating an additional registration step (likely a signed/marketplace-side identifier) is required and is not user-controllable in v1.0.6. |
| `agy plugin install <target>` for an arbitrary local path | Requires a `plugin@marketplace` reference; no documented way to point it at a freshly-generated local plugin dir for a one-shot per-terminal MCP server. |

The functional consequence: there is no way for CAO to register a per-terminal MCP server (with a unique `CAO_TERMINAL_ID` env var) before launching `agy` and have the running model actually see and call its tools.

### What was investigated

1. **`agy --help` / `agy help <subcommand>`** — no MCP flag or subcommand.
2. **`~/.antigravity/`** — Electron IDE config only; no MCP config.
3. **`~/.gemini/antigravity-cli/`** — CLI config directory. Contains `settings.json`, `plugins/`, `mcp/` (per-server cached tool schemas), `import_manifest.json`, `builtin/`, `implicit/`. The `mcp/` directory enumerates *built-in* MCP servers shipped with `agy` (e.g. `sequential-thinking`, `visualization`, `google-cloud-resource-manager`); these are not user-extensible.
4. **`agy plugin list` vs. on-disk `import_manifest.json`** — they disagree. The on-disk file lists 6+ historically-imported plugins, but `agy plugin list` shows only the subset that came from canonical sources (`gemini`, `claude`). This confirms the registration path is not just a file write.
5. **Direct `mcpServers` in `~/.gemini/antigravity-cli/settings.json`** — written, then asked the running model to enumerate accessible MCP servers. Result: the synthetic entry did not appear in the model's tool list. Confirms `agy` does not read this key the way `gemini` reads its own settings.json.
6. **`agy plugin install` / `agy plugin import`** — both require either a canonical source (`gemini` / `claude`) or a marketplace reference. Neither accepts an ad-hoc local plugin path suitable for per-terminal injection.

### Conditions that would unblock v0.1

Any one of the following Antigravity-side changes would make MCP integration viable:

- An `agy mcp add` / `agy mcp remove` subcommand pair (mirroring `gemini mcp add`).
- An `mcpServers` key in `~/.gemini/antigravity-cli/settings.json` that `agy` reads at startup, identical to gemini's settings.json schema.
- A documented `agy plugin install <local-dir>` form (or equivalent `--scope=user --source=local` flag) that accepts a freshly-generated plugin directory without going through a marketplace.
- A documented environment variable like `AGY_MCP_CONFIG` that points `agy` at an external JSON file with the standard MCP server shape.

When one of these ships, the provider can be upgraded to write a per-terminal MCP config file mirroring `gemini_cli.py`'s `_register_mcp_servers` / `_unregister_mcp_servers` pair (the file shape is already identical — `{"mcpServers": {name: {command, args, env}}}` — only the registration call site needs to change).

### Implementation note

`AntigravityCliProvider._build_agy_command()` currently logs a warning and proceeds when `profile.mcpServers` is non-empty. This is the documented chat-only v0 behavior — it is not a bug.

## Tool restrictions

Not implemented in v0. `agy` does not expose a documented tool deny-list mechanism (no `Policy Engine` equivalent like `gemini`'s `~/.gemini/policies/*.toml`). Profiles that set `allowedTools` to anything other than `*` will see a one-line warning at terminal-create time; the terminal will run with `agy`'s default tool surface.

Likely to land alongside the MCP work in v0.1 — when `agy` ships a tool-restriction surface, it can be wired into `_build_agy_command` the same way `gemini_cli._write_policy_deny_rules()` is.

## Known limitations

- Regex patterns in `antigravity_cli.py` were calibrated from a single 220×50 tmux capture session on macOS arm64 with `agy 1.0.6`. Lines marked `# CALIBRATE` should be re-derived from larger fixture sets before any provider upgrade.
- The trust-folder prompt (`"Do you trust the contents of this project?"`) is auto-confirmed with Enter on first launch in an unknown cwd.
- `agy`'s response area does not use a special anchor character (no `✦` like Gemini); responses are 2-space-indented plain text. Tool-call rendering for built-in tools may use different indents; this has not been exhaustively characterised.
