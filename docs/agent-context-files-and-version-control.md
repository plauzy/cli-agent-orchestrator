# Agent Context Files and Version Control

> **Status:** Advisory / decision record. Scope: how this repository should treat
> agent-oriented context files — `AGENTS.md`, the `.kiro/` tree, `.agents/`,
> `.claude/`, and `.codex/` — with respect to Git tracking.

## TL;DR recommendation

**Do not blanket-`.gitignore` all agent dotfiles.** Split them by *what the content
actually is*, not by the fact that a tool created it:

| Path | Track in Git? | Why |
|------|---------------|-----|
| `AGENTS.md` (repo root) | **Yes** | Durable, human-authored "README for agents". Already tracked today. |
| `.kiro/steering/` (except generated files) | **Yes** | Team-shared, durable rules/standards. Reviewable, benefits everyone. |
| `.kiro/specs/` | **Yes (recommended)** | Durable design artifacts (requirements/design/tasks) worth reviewing in PRs. |
| `.kiro/steering/cao-memory.md` | **No** | Machine-generated by CAO's memory plugin; regenerated per run. |
| `.kiro/settings/` (e.g. `mcp.json`) | **No** | May carry secrets/tokens and per-developer endpoints. |
| `.agents/` | **No** (default) | Tool/runtime state; local unless it demonstrably holds authored, shareable context. |
| `.claude/`, `.codex/` | **No** | Per-developer tool state; in this repo `.claude/CLAUDE.md` is machine-injected. |

The principle: **track durable, human-authored, shareable indexing context; ignore
machine-generated, per-developer, ephemeral, or secret-bearing tool state.**

## Why this question is non-trivial in *this* repo

CAO is not just a *consumer* of these files — it *writes* them. The built-in memory
plugins inject an up-to-date workspace/memory context block into provider-specific
locations on terminal creation:

- **Kiro CLI** → `<cwd>/.kiro/steering/cao-memory.md`. Kiro natively loads every
  `*.md` under `.kiro/steering/`. The plugin **owns this file whole** and overwrites
  it on each run (no in-file markers).
- **Claude Code** → `<cwd>/.claude/CLAUDE.md`. The plugin owns only a **delimited
  block** (`<!-- cao-memory:begin -->` … `<!-- cao-memory:end -->`) and preserves
  surrounding hand-written content.
- **Codex** → `<cwd>/AGENTS.md` (repo root). Codex CLI reads `AGENTS.md` from the
  working directory as project instructions. `AGENTS.md` is a **user-authored,
  repo-root file** (the "README for agents"); the plugin owns only the same delimited
  block and preserves all surrounding hand-written content.

So the same directory can simultaneously hold **content a team wants to review and
share** (authored steering rules, `AGENTS.md` prose, specs) and **content that churns
on every run or is developer-local** (the injected `cao-memory` block, the whole-file
`cao-memory.md`, local `settings/`). A blanket ignore throws away the former to avoid
the latter; committing everything creates noise and risks leaking secrets.

## Current state (as of this branch)

`.gitignore` treats all four dotfile directories as opaque "AI tool configs":

```gitignore
# AI tool configs
.claude/
.codex/
.agents/
.kiro/
```

Consequences today:

- `AGENTS.md` at the repo root **is tracked** (it is a file, not under an ignored dir).
- `.kiro/steering/` and `.kiro/specs/` are **not tracked** — team steering rules and
  design specs (like this one) can only be committed with `git add -f`, which is easy
  to forget and invisible in normal `git status`.
- Everything under `.claude/`, `.codex/`, `.agents/` is ignored, including any
  authored content someone might place there.

This is a reasonable *default* (it prevents accidental secret/noise commits), but it is
too coarse for a project that deliberately produces shareable steering and spec content.

## The taxonomy that drives the decision

Classify each file along three axes:

1. **Authorship** — human-authored vs machine-generated.
2. **Audience/lifetime** — durable & team-shared vs ephemeral & per-developer.
3. **Sensitivity** — safe to publish vs secret-bearing (tokens, endpoints, PII).

Track a file only when it is **human-authored (or intentionally curated), durable and
team-shared, and safe to publish.** If it fails any axis, ignore it.

| File / dir | Authorship | Lifetime | Sensitivity | Verdict |
|------------|-----------|----------|-------------|---------|
| `AGENTS.md` (prose) | Human | Durable | Safe | Track |
| `AGENTS.md` (injected `cao-memory` block) | Machine | Ephemeral | Usually safe | Keep out of commits (see below) |
| `.kiro/steering/*.md` (authored) | Human | Durable | Safe | Track |
| `.kiro/steering/cao-memory.md` | Machine | Ephemeral | Usually safe | Ignore |
| `.kiro/specs/**` | Human (agent-assisted) | Durable | Safe | Track |
| `.kiro/settings/mcp.json` | Human | Durable | **Secret-bearing** | Ignore |
| `.claude/CLAUDE.md` | Machine block + human | Mixed | Usually safe | Ignore (repo convention) |
| `.codex/**` | Machine | Ephemeral | Mixed | Ignore |
| `.agents/**` | Tool | Ephemeral | Mixed | Ignore (default) |

## Recommended `.gitignore` changes

Replace the coarse block with a selective one. The pattern uses `.kiro/*` (ignore
*children* of `.kiro`, not the directory itself) so that negations for specific
subdirectories work:

```gitignore
# AI tool configs — per-developer / ephemeral tool state (ignored)
.claude/
.codex/
.agents/

# Kiro: ignore everything under .kiro by default...
.kiro/*
# ...but TRACK durable, team-shared context:
!.kiro/steering/
!.kiro/specs/
# ...then RE-IGNORE machine-generated or secret-bearing content inside them:
.kiro/steering/cao-memory.md
.kiro/settings/
```

Notes:

- Using `.kiro/*` rather than `.kiro/` is what makes `!.kiro/steering/` and
  `!.kiro/specs/` effective — Git cannot re-include a path if its parent directory is
  ignored wholesale.
- After changing `.gitignore`, previously force-added spec files will show up normally
  in `git status`; no more `git add -f`.
- If you prefer to keep specs out of the repo, drop the `!.kiro/specs/` line — but
  tracking them makes design decisions reviewable in PRs, which is the point of the
  spec workflow.

## Handling the injected `cao-memory` block in `AGENTS.md`

`AGENTS.md` is tracked, but CAO injects a delimited block into it. Two clean options:

1. **Convention (simplest):** treat the region between
   `<!-- cao-memory:begin -->` and `<!-- cao-memory:end -->` as local-only. Don't stage
   changes limited to that block. Good enough for small teams.
2. **Git clean filter (robust):** strip the block automatically on staging so the
   committed `AGENTS.md` only ever contains human prose. Add to `.gitattributes`:

   ```gitattributes
   AGENTS.md filter=strip-cao-memory
   ```

   and configure the filter (each developer runs this once, or ship it via a bootstrap
   script):

   ```bash
   git config filter.strip-cao-memory.clean \
     "sed '/<!-- cao-memory:begin -->/,/<!-- cao-memory:end -->/d'"
   git config filter.strip-cao-memory.smudge cat
   ```

   This keeps the working copy (what the agent reads) complete while the committed
   version stays free of generated churn.

The whole-file `.kiro/steering/cao-memory.md` needs no filter — it is simply ignored.

## Security considerations

- **`.kiro/settings/mcp.json` can contain secrets** (tokens, provider endpoints).
  Keep it ignored. Commit a redacted `mcp.json.example` instead if you want to share
  structure.
- Machine-generated memory/index blocks can inadvertently capture local paths,
  hostnames, or snippets of private context. Prefer ignoring them; if you must track
  one, review it like any other artifact.
- Never rely on `.gitignore` alone for secrets that already exist in history — rotate
  the secret and scrub history if a token was ever committed.

## Migration steps

1. Apply the `.gitignore` changes above.
2. `git rm -r --cached` any now-untracked-by-policy files that were previously
   force-added and should not be tracked (e.g. a stray `cao-memory.md`).
3. `git add .kiro/steering/ .kiro/specs/` to bring durable context under normal
   tracking (no `-f` needed after the ignore change).
4. (Optional) Add the `.gitattributes` clean filter for `AGENTS.md`.
5. Add a short note to `CONTRIBUTING.md` pointing here so the policy is discoverable.

## Summary

Blanket-ignoring agent dotfiles is a safe default but the wrong long-term policy for a
repository that produces shareable steering rules and specs. Track the durable,
human-authored, publish-safe context (`AGENTS.md`, `.kiro/steering/`, `.kiro/specs/`)
and ignore the machine-generated, per-developer, or secret-bearing state
(`cao-memory` outputs, `.kiro/settings/`, `.claude/`, `.codex/`, `.agents/`). This
gives agents up-to-date indexing context locally while keeping the committed history
clean, reviewable, and secret-free.
